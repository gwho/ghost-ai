# Feature 05 — Prisma Schema and Data Layer

## What This Feature Does

This feature wires in the real database layer for Ghost AI. Before this, projects existed only as mock data in memory. After this, the app has:

- A **PostgreSQL schema** defining `Project` and `ProjectCollaborator` tables
- A **Prisma client singleton** (`lib/prisma.ts`) that any server code can import to run queries
- A **live migration** applied to the Prisma Postgres cloud database

---

## Why We Did It This Way

### Multi-file Prisma schema (`prisma.config.ts` + `prisma/models/project.prisma`)

Prisma 7 supports splitting your schema across multiple `.prisma` files. We configure it in `prisma.config.ts` at the project root:

```ts
export default defineConfig({
  schema: path.join('prisma'),   // tells Prisma to read all .prisma files in this folder
  datasource: { url: env('DATABASE_URL') },
})
```

**Why the datasource is in `prisma.config.ts` and not `schema.prisma`:** Prisma 7 removed the `url` field from `datasource` in schema files. The connection URL now lives exclusively in the config file. This enforces a clean separation: the schema describes the shape of your data, the config handles environment-specific wiring.

### Singleton with URL branching (`lib/prisma.ts`)

The singleton checks which type of database connection to use at startup:

```ts
if (url.startsWith('prisma+postgres://')) {
  return new PrismaClient({ accelerateUrl: url })  // Prisma Accelerate (caching + pooling)
}
const adapter = new PrismaPg({ connectionString: url })
return new PrismaClient({ adapter })               // Direct Postgres via pg driver
```

**Why two paths?** Prisma Accelerate (URLs starting with `prisma+postgres://`) provides edge-friendly connection pooling and query caching. Direct `postgres://` URLs use the `@prisma/adapter-pg` driver adapter, which is the standard Node.js pg driver. The current `DATABASE_URL` is a direct Postgres URL on Prisma's cloud infra, so the adapter path runs.

**Why `globalThis` for the singleton?** In Next.js development, the server module is hot-reloaded on every file change. Without the global cache, each reload would open a new database connection. Caching the client on `globalThis` means we reuse the same connection across hot reloads.

### Generator output at `lib/generated/prisma`

Prisma 7 generates TypeScript types to `node_modules/.prisma/client/` by default. TypeScript's `moduleResolution: "bundler"` (used by Next.js) treats the re-export path `.prisma/client/default` as a RELATIVE path from `@prisma/client/`, and can't find the generated types. Node.js itself treats `.prisma` as a bare specifier and resolves it correctly at runtime, but TypeScript disagrees.

**Fix:** Set an explicit `output` in the generator so the types land in a project-owned directory TypeScript can always find:

```prisma
generator client {
  provider = "prisma-client-js"
  output   = "../lib/generated/prisma"
}
```

Then import from `@/lib/generated/prisma` in `lib/prisma.ts`. The generated folder is committed to git (or regenerated by `prisma generate` in CI).

## Files Changed

| File | What it does |
|---|---|
| `prisma.config.ts` | Prisma CLI config — sets schema folder, loads DATABASE_URL from `.env.local`, and routes connections |
| `prisma/schema.prisma` | Datasource provider + generator (output path) — no models here |
| `prisma/models/project.prisma` | `Project` and `ProjectCollaborator` models with indexes and cascade delete |
| `lib/prisma.ts` | Exported `prisma` singleton, cached on `globalThis` |
| `lib/generated/prisma/` | Prisma-generated TypeScript client (run `npx prisma generate` after schema changes) |
| `prisma/migrations/` | SQL migration files applied to the database |

## Data Model

```
Project
├── id             (cuid, PK)
├── ownerId        (Clerk user ID)
├── name
├── description?
├── status         (DRAFT | ARCHIVED)
├── canvasJsonPath? (Vercel Blob URL, added later)
├── createdAt
├── updatedAt
└── collaborators  → ProjectCollaborator[]

ProjectCollaborator
├── id
├── projectId  → Project (CASCADE DELETE)
├── email
└── createdAt
```

Indexes: `ownerId`, `createdAt` on Project; `email`, `(projectId, createdAt)` on ProjectCollaborator. Unique constraint on `(projectId, email)`.

## How to Use the Client

```ts
import { prisma } from '@/lib/prisma'

// Create a project
const project = await prisma.project.create({
  data: {
    ownerId: userId,
    name: 'My Architecture',
  },
})

// List projects for a user
const projects = await prisma.project.findMany({
  where: { ownerId: userId },
  orderBy: { createdAt: 'desc' },
})
```

## Reusable Lessons

1. **Prisma 7 moved datasource URL out of schema files.** The URL belongs in `prisma.config.ts`, not `schema.prisma`.

2. **TypeScript `moduleResolution: "bundler"` and Prisma's default output path are incompatible.** Always set an explicit `output` in the generator when using Next.js.

3. **Hot-reload protection for DB connections:** Cache the client on `globalThis` in development. The `NODE_ENV !== 'production'` guard ensures this only happens locally.

4. **The `.prisma` module resolution quirk:** Node.js CJS treats `.prisma/...` as a bare specifier (node_modules lookup) while TypeScript bundler treats it as a relative path. When you hit a "no exported member" error after `prisma generate`, this is usually why.
