# Spec Explanation: Feature 07 — Wire Editor Home to Real Project API

## What This Feature Does

Until now the sidebar showed fake projects — a hardcoded list that lived only in the browser's memory and disappeared on every refresh. This feature replaces that with real data: projects are fetched from the database when the page loads, and every create/rename/delete action actually talks to the API we built in Feature 06.

## The Core Problem: Where Does Data Come From?

In Next.js there are two kinds of components:

- **Server components** run on the server. They can read the database directly. They cannot use browser APIs like `useState` or `onClick`.
- **Client components** (`"use client"`) run in the browser. They can use state and handle events. They cannot call the database directly.

The sidebar needs data from the database, but it also needs to react to clicks and dialog state. That tension is what drives most of the structural changes in this feature.

## Why Did the Layout Need to Change?

The old `app/editor/layout.tsx` had `"use client"` at the top. That meant it was a client component — it could not fetch from the database.

The fix is a two-step split:

1. **`app/editor/layout.tsx`** becomes a server component. It fetches the project list and passes it as props to a child.
2. **`components/editor/editor-shell.tsx`** (new, `"use client"`) receives those props and handles all the browser interactivity — the sidebar open/close toggle, the dialog context provider, etc.

This pattern — server component fetches, client component acts — is the standard Next.js App Router approach.

## What Is `lib/project-data.ts`?

This is a server-side helper function called `getEditorProjects()`. It runs only on the server (never in the browser) and returns two lists:

- `owned` — projects you created
- `shared` — projects where you were added as a collaborator

For owned projects it queries by your Clerk user ID. For shared projects it needs your email address — the `ProjectCollaborator` table stores collaborators by email, so we fetch the email from Clerk's `currentUser()` first.

Both database queries run at the same time with `Promise.all` so the page doesn't wait for one before starting the other.

## What Is `useProjectActions` and Why Replace the Old Hook?

The old `useProjectDialogs` hook had fake mutation handlers that just waited 400ms and closed the dialog. The new `useProjectActions` hook does the same job — managing dialog open/close state and triggering mutations — but calls the real API:

| Action | API Call | Then |
|---|---|---|
| Create | `POST /api/projects` | Navigate to `/editor/{newId}` |
| Rename | `PATCH /api/projects/{id}` | Refresh sidebar |
| Delete | `DELETE /api/projects/{id}` | Redirect home (if on that workspace) or refresh |

The hook accepts the initial project list as arguments (passed down from the server fetch) and holds them in local `useState`.

## The "State Sync" Problem

Here is a subtle issue: when `router.refresh()` tells Next.js to re-fetch the server components, the layout runs again and passes fresh project data as new props to the `EditorShell`. But React's `useState` only initializes from its initial value once — it does not automatically update when props change.

To fix this, the hook watches for changes in the props using `useEffect`:

```ts
const ownedKey = initialOwned.map((p) => p.id).join(",")
useEffect(() => {
  setOwnedProjects(initialOwned)
}, [ownedKey])
```

Instead of watching the array itself (which would fire on every render because JavaScript always creates a new array object), it watches a stable string of the project IDs. When that string changes — meaning the server returned a different set of projects — the local state updates.

## The Create Dialog Slug Preview

When you type a project name, the dialog shows a preview like `ghost.ai/my-project-ab3f`. That preview is made of two parts:

- **Slug** — the name converted to lowercase with spaces replaced by hyphens (`My Project` → `my-project`)
- **Suffix** — a short random string generated fresh each time you open the create dialog

The suffix makes each workspace URL unique even if two projects have the same name. It is only for the preview display — the actual project ID is a cuid generated by the database.

"The project ID and Liveblocks room ID should stay aligned" means: when Liveblocks is wired later, the room name will just be the project's database ID. No separate field needed.

## Why Does the Home Page Button Need Its Own Component?

The home page (`app/editor/page.tsx`) is now a server component. But the "New Project" button needs to call `openCreate()` — a function from React context, which is a browser concept.

Server components cannot use context. So the button is extracted into a tiny client component (`editor-home-actions.tsx`). The server page renders the static text, then drops in `<EditorHomeActions />` for the interactive part. This keeps the page itself server-side while still supporting the button.
